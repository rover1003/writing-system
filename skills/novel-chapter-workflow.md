# 小说章节修改工作流

## 核心原则

- **小改（一两句话/个别段落）**：直接用 terminal `patch`，不调MCP
- **大改（结构/新增情节）**：调MCP `revise_chapter`
- **用户明确说"只改结尾/只改某段"**：只改那个位置，不动其他内容
- **我是调度者，用户是指挥者**：小改用 terminal `patch` 直接执行，不问用户要具体文本，也不让用户执行操作

## 标准流程（必须遵守！）

**初稿 → 顾问审查 → 汇报审查结果 → 等用户确认修改点 → revise → 用户审阅revise版 → 用户说"定稿" → finalize**

### Prompt 注入顺序（write_chapter 为例）

| 顺序 | 内容 | 来源 | 范围 |
|------|------|------|------|
| 1 | 【分级大纲】 | XML大纲 | 当前章PlotPoint |
| 2 | 【最近章节正文】 | `load_recent_chapters()` | 最近3章全文 + 更早章节各600字预览 |
| 3 | 【前情摘要合并版】 | `load_prequel_summary()` | 所有已定稿章节摘要 |
| **4** | **【RAG 召回片段】** | `search(outline[:800])` | **语义检索top_k=4，不限章节** |
| 5 | 【世界观与人物设定】 | XML | world-building规则+口吻规范 |

**重叠分析**：RAG召回可能与第2/3项重叠（如RAG召回了近期章节片段），但无害——LLM读到重复内容只会多花token，不会出错。

**RAG的独特价值**：捕捉「跨章节弱线索」，尤其是跨越很远的情节（前3章全文+摘要均不覆盖）。

#### ⚠️ 远程通讯机制检查

**正确逻辑**：远程通讯依赖特定设备走卫星信号，不走地面网络。通讯方式的描写必须与设定一致。

**强制流程**：任何涉及角色远程联系的情节 → 写之前先确认通讯机制设定 → 不得凭"电话"常识自行演绎

#### ⚠️ 禁止元叙述连接句

典型错误：用章节号作为正文的连接句（如"第一章，撞击前大概两个月"）。

正确写法：用自然的时间/场景回忆句式，如"那是撞击前大概两个月的事……"

**禁止词汇**："第X章"、"上一章"、"本章"等章节指代词出现在正文里

#### ⚠️ 流量/下载常识——断网后没有地方下载

正确逻辑：
- 断网=所有地面服务器全部宕机
- 所有音频/视频/文档必须来自**本地预存**或**灾前下载缓存**
- 设备走卫星信号，不走地面服务器

**禁止写法**：
- ❌ "连夜下载"、"网上搜的"、"找到的音频"
- ✅ 正确来源：设备自带音频库、灾前下载到本地的文件、亲自录音

#### ⚠️ 物资数量一致性检查——写之前必须grep已定稿章节

**强制流程**：
1. 写涉及物资数量的情节之前 → 先 grep 确认实际剩余数量
2. 特别检查：主角"给了/送了/换了"物资的情节 → 必须在已定稿章节中找到原文验证
3. 如果是新增物资（原大纲没写）→ 先问用户"囤了多少"，不能凭常识编

**核心原则**：写回忆/引用其他章节具体细节（人名/地名/物品/事件）之前，必须 `read_file` 那个章节的原文，验证：

1. 这个名字/细节在原文里**确实存在**
2. 具体措辞是什么（避免同义改写后产生歧义）
3. 上下文是否支持这个引用的方向
4. 联系的方式/时间是否符合已写细节

**例外**：如果用户明确说"这个人物/细节在人物设定/世界观设定中一直都有"，以用户确认为准。

**写作禁区**：
- ❌ 不得在前文没有铺垫的情况下凭空创造跨章节关联的角色名
- ❌ 不得用"第一章""第X章"这类元叙述词汇作为正文的连接句
- ❌ 用"后来我才知道，他的名字叫XX"来揭示名字——昵称本来就能看到，这样写逻辑不通
- ✅ 正确写法：回忆时自然带出名字，如"他的微信昵称叫'XX'，头像是个默认灰色小人"，在同一句话里完成回忆和命名，不分两次写

#### ⚠️ 对话逻辑链——某角色突然提出话题必须有铺垫

**正确逻辑链**：
- 主角先问路线/怎么找这里
- 对方提到"标记"作为找路的方式
- 主角起疑，反问"什么标记"
- 对方才说出标记的事
- 主角联想到自己这边也有

**禁止令**：禁止任何角色在对方没有提及的情况下主动抛出关键信息。信息必须一层一层递进，不能跳步。

#### ⚠️ 虚拟地名检查——不得残留真实地名

**强制流程**：章节写完后，grep检查是否有真实地名残留，全部替换为虚拟地名。

#### ⚠️ 灾后场景常识——禁止保留灾前基础设施假设

**禁止词汇/场景**：
- ❌ 动车、火车（铁路系统已瘫痪）
- ❌ 服务区、加油站（能源短缺）
- ❌ 跟着运输车走（运输系统已停运）
- ✅ 走国道/省道/小路，避开检查站
- ✅ 开一到两天，带备用油桶

**检查方法**：涉及交通/通讯/能源的情节时，问自己"这个在灾后还能存在吗？"

#### ⚠️ 写章节前必查"情节是否已在上章写过"

**强制流程**：写本章之前，先grep最近3章相同类型情节：
- 如果同类情节已在近期章节**完整写完**，本章必须跳过该情节的铺垫阶段，直接切到**消化结果的下一阶段**

**大纲与已写章节情节重叠的处理流程**：
1. `read_file` 大纲，找到重叠情节在哪个未来章节
2. `read_file` 已写章节全文，找到重叠内容**精确行号范围**
3. 确认删除后上下文能否衔接（一般需要一句话过渡）
4. 向用户说明：删什么、留什么、删后怎么衔接
5. 等用户确认后再执行patch

#### ⚠️ 时间线检查：先查前章结尾再写本章开头

**强制流程**：
1. `grep -n "Day" 最近3章` 找出所有已定稿章节的Day标记，建立连续时间链
2. 如果本章前已有N章，必须确认第N章的Day标记，才能写第N+1章
3. **不得只查前章1章就下结论**——必须查整个链
4. 写本章时间标记前，grep大纲确认本章大纲写的Day数字，两方对照
5. 三方不一致 → 以**已定稿章节正文**为准，修正大纲

#### ⚠️ 维护类请求先grep再操作

**强制流程**：
1. 用户说"把XX加到YY表/伏笔/设定"
2. **先grep目标文件**确认该条目是否已存在
3. 已存在 → 直接报告用户"已在XX位置，无需添加"；不存在 → 再执行添加

步骤详解：
1. `write_chapter` → 报告用户（**不要跳过审查**）
2. 调用 `review_chapter` → 顾问审查
3. 报告用户审查结果，**等用户明确说"确认，调用MCP修改"**
4. 用户确认后 → 调用 `revise_chapter`
5. revise后检查输出末尾是否有"未完待续"，有则patch删掉
6. **检查MCP revise输出是否与本地文件内容一致**：
   - **如果内容一致**：直接patch删"未完待续"即可
   - **如果内容不一致（完全新版）**：必须向用户确认"当前有两个版本，使用新版还是保留旧版？"，等用户指令后再写入文件覆盖

7. **revise后立即clean再写入（新文件场景）**：
   - revise返回的是内存中的LLM输出字符串（不含"未完待续"），不是从磁盘读的
   - 若目标章节文件尚不存在（首次写入），**先从revise输出末尾patch删"（未完待续）"，再write_file**
   - 不要先write_file再patch——磁盘文件不存在时patch会失败
   - 验证顺序：MCP返回 → patch删"未完待续" → write_file → 确认文件写入

7. **把revise版正文呈现给用户，等用户审阅**
8. **用户确认没问题，说"定稿"后** → 调用 `finalize_chapter` 定稿
9. 执行定稿后对齐清单（章节进度总表/伏笔表/XML大纲/mem0）

### 🚨 强制检查点（违反即停）

**写完初稿后，在调用顾问审查之前，禁止执行以下任何操作：**
- ❌ 直接调用 `finalize_chapter`
- ❌ 直接调用 `revise_chapter`
- ❌ 跳过审查直接向用户报告"可以直接定稿"
- ❌ 将revise版写入文件前未告知用户
- ❌ 用户未审阅revise版就自行finalize

**正确做法永远是：写 → 审 → 判断 → 改（一次合并）→ 写入文件 → 用户审阅 → 用户说定稿 → 定。**

## patch 删内容的规范

**错误做法**：只删几句话，漏掉周围段落。

**正确做法**：
1. 先 `read_file` 确认要删的内容的**精确行号范围**
2. patch 时 old_string 要足够长，确保唯一匹配
3. 删完后读文件验证

## 修改后验证清单

1. 读文件确认修改生效
2. 检查上下文衔接是否断裂
3. 确认没删多/没留多

## 定稿后主动汇报规则

**强制行为**：定稿后，**不等待用户追问**，主动在同一条消息里汇报：
1. `✅ 章节进度表` — 已更新（列出章节号和状态变化）
2. `✅ 前情摘要` — 已写入（如MCP未自动处理）
3. `✅ XML物资表` — 如涉及物资变化，已同步
4. `✅ RAG索引` — 已更新（如已执行）

如果某步骤因故未执行，明确说明"❌ RAG索引 — 待执行（原因）"。

**禁止**：finalize后只说"已定稿"就等用户追问。

## 定稿后对齐清单（必须逐项执行）

| 步骤 | 操作 | 说明 |
|------|------|------|
| 1 | 章节进度总表 → ✅定稿 | 更新章节状态 |
| 2 | 章节统计总览 → 已定稿数+1、字数更新 | 更新统计 |
| 3 | 最后更新日期 | 更新元信息 |
| 4 | 前情摘要合并版 → 包含新章节摘要 | 追加新章节摘要 |
| 5 | 伏笔档案 → 已回收/已埋设 | 更新伏笔状态 |
| 6 | 人物设定库 → 新人物已添加 | 新增人物入库 |
| 7 | 物资消耗记录 → 已更新（如涉及） | 物资数量变化 |
| 8 | 章节标题 → ≥2汉字 | 正文标题行 |
| 9 | RAG 索引 → 为本章生成向量块并加入索引 | 更新向量索引 |

**步骤9 RAG 索引详细操作**（finalize 后必须执行）：
1. 用 `execute_code` 调用 `rag.py` 的 `index_chapter(chapter_num, chapter_text)` 为本章生成向量块
2. 追加写入 `.mcp/rag_index.json`
3. 验证：`execute_code` 调用 `len(rag_index["chunks"])` 确认总数增长

### ⚠️ 更新进度表前的强制核验步骤

**步骤**：
1. 更新章节进度总表（步骤1）之前，**先grep确认实际✅定稿数量**
2. 对照确认：grep `✅定稿` 数量 = 声称的已定稿数
3. 如果发现不匹配，**先修正表格行**，再更新统计总览

### ⚠️ 批量patch章节行时的防丢失法则

**正确做法**：
- 批量修改相邻章节行时，**每次最多2行**一起patch
- 每patch一次，立即grep验证目标行存在
- 如果一次要改N行（N>2），**从后往前**逐对修改，不要合并为一次大patch

### 章节标题硬性规定
- **标题必须≥2个汉字**
- 风格要求：**简洁明了**，把当前章节最核心的情节点出来即可
- 检查方式：标题中去掉数字和下划线后，汉字数≥2

## 分工内容处理原则

如果当前章节已经讨论过团队分工，下一章大纲里要把这段分工**备份为备注**，不要再重复写。

## patch 精准删内容的步骤（精简章节用）

大量精简时（删5+段重复/冗余），建议顺序：

1. **先读全文**（`read_file` offset=1 limit=全文），掌握整体结构
2. **边读边记**精简点（标注"删哪段、为什么删"）
3. **每patch 2-3段就读一次**验证，防止连锁失败
4. patch时 old_string 要足够长，**覆盖到上下文唯一锚点**

### patch 失败处理（Found N matches）

- **Found 2+ matches**：立即停手，**不要replace_all**
- 原因：小说正文常有重复句式
- 正确做法：`read_file` 定位到该段落，用**更大上下文**（前后各1-2句）重新拼old_string

### 🚨 角色凭空出场检查（扩团章节必查）

**强制流程**：新角色首次出现 → grep前5章确认该角色是否已在队伍中 → 如无，插入"主角观察+来历说明"锚定段落

### 🚨 群聊戏份记忆混淆检查

**强制流程**：写群聊场景前 → grep该群最近发言记录 → 确认每条消息的发送者和时间点 → 三要素（发送者+时间+事件）缺一不可

### 小改 patch vs MCP 的判断标准（重要！）

**terminal patch 适用的场景**：
- 改写/删除/调整已有的**少量文字**（1~3句）
- 时间线标记修正（如加 `## Day N`）
- 禁用词/逻辑错误/错别字
- 删除重复段落

**必须用 MCP revise_chapter 的场景**：
- **添加新的情节要素**（即使只有1~2段）——因为需要调整周围文字才能自然衔接
- 需要改写较大段落才能让新内容融入
- 用户说"把这个情节加回去"——这意味着**新增内容**，不是修改已有文字

**判断技巧**：
- 用户说"XX情节没有了，应该加上" → MCP（新增）
- 用户说"XX台词写错了，改成YY" → terminal patch（改写）
- 时间线错位+需要加新伏笔 → MCP（结构调整）

### MCP revise 返回全章新版时的处理

**现象**：MCP revise 后返回的内容行数/结构可能与本地文件完全不同。

**正确步骤**：
1. 先让用户看过新版内容
2. 用户说"写入"后再 write_file 覆盖

### 时间线修复的标准流程

**发现问题**：用户说"时间线不对"

**正确步骤**：
1. 查前章结尾的 Day 标记
2. 查本章所有时间标记（grep Day/早上/下午/晚上）
3. 查下章大纲确认正确 Day 编号
4. 三方对照找出错位点
5. 用 `## Day N` 标记 + 分节前时间锚定句修正

### 本章学到的坑（精简章节）

1. 精简前先通读——章节里重复句式很多，要用足够长的上下文锚定
2. 删分工段：小说正文"大家各自散开"也出现多次，要用足够长的上下文锚定
3. 并排删多段：先用一句话过渡，再删掉冗余的分述段落
4. 合并多个段落时，确保关键对话信息不丢
