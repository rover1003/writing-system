# 长篇小说章节写作标准工作流

> 核心原则：**写 → 审 → 改 → 用户审阅 → 定稿**，每步不可跳过，全程使用 MCP 工具

---

## 一、触发条件

| 用户指令 | 执行流程 |
|---------|---------|
| "准备写第X章" / "开始写第X章" | 标准流程8步 |
| "审第X章" / "改第X章" / "定稿第X章" | 完整流程 |

**禁止**：
- ❌ 自己写正文（必须调用 `mcp_writer_write_chapter`）
- ❌ 自己审稿（必须调用 `mcp_consultant_review_chapter`）
- ❌ 跳过顾问直接 finalize
- ❌ 用户未审阅 revise 版就自行 finalize

---

## 二、标准流程（8步）

### 第1步：并行读取设定

启动 subagent 并行读取：

**subagent #A — 大纲 + 时间线**
- 分卷大纲对应章节标签
- 设定库 `foreshadowing` 相关标签（如涉及伏笔）

**subagent #B — 人物 + 世界观**
- 设定库 `personas` + `writing-rules`
- 设定库 `chat-groups`（群聊场景必读）

**subagent #C — 前几章正文（情节衔接）**
- 最近 10 章已定稿正文

### 第2步：拟大纲初稿（仅新写需要）

根据大纲中的 PlotPoint 生成结构化分级大纲。

### 第3步：顾问审查大纲

调用 `mcp_consultant_review_outline`，按顾问反馈修正。

### 第4步：写正文初稿

**必须调用**：`mcp_writer_write_chapter(chapter_num, outline)`

### 第5步：并行读取设定（审查前置）

同第1步，确保上下文完整。

### 第6步：逐段审查

| 检查项     | 说明               |
| -------- | ---------------- |
| 人设口吻    | 对话是否符合人物设定     |
| 情节连贯性   | 事件发展是否符合因果链    |
| 时间线逻辑   | 关键节点是否自洽，与前3章衔接 |
| 物资数量    | 消耗/囤货数量与前文是否一致  |
| 伏笔      | 前文伏笔是否正确回收     |
| 禁用词     | 去 AI 味，禁用"未完待续"等 |

### 第7步：顾问二次审查

调用 `mcp_consultant_review_chapter(chapter_num, text, stage=review)`

### 第8步：向用户汇报

---

## 三、顾问审查注入规则（强制）

调用 `review_chapter` 前必须注入：
```
【本章大纲核心 PlotPoint】（从大纲读取）
【地名虚拟化对照表】（如有）
【前3章已出现的群ID / 物资细节】（如有）
```

**顾问可能出错**：建议中出现具体人名/地名/道具名时，必须先 grep 验证该要素是否存在于当前章节，不存在则跳过。

---

## 四、顾问审查返回后处理

| 问题类型             | 处理方式                                   |
| ---------------- | -------------------------------------- |
| 同向修改意见（全需改）    | 直接调用一次 `revise_chapter`，所有问题合并发出       |
| 多方案含义不同         | 向用户说明，等用户选择                           |
| 顾问建议含不存在的人名/地名/道具 | grep 验证，跳过不存在项，向用户说明               |

**执行后仍必须等用户审阅正文才能 finalize。**

---

## 五、修改工作流

| 场景              | 方式                              |
| --------------- | ------------------------------- |
| 改写 / 删除 1-3 句    | terminal `patch`（直接执行，不调 MCP）        |
| 用户说"只改结尾"      | terminal `patch`（直接执行，不调 MCP）        |
| 加新情节（当前文件没有的） | `revise_chapter`                      |
| 时间线修正 + 新增伏笔    | `revise_chapter`                      |

**判断口诀**：文件里没有这情节 = 新增 → `revise_chapter`；文件里有这情节 = 改写 → `patch`

### revise 后必须操作

1. 检查输出末尾是否有"未完待续"→ patch 删掉
2. 将 revise 版正文写入文件
3. 向用户展示修改摘要，请用户审阅

### patch 删内容规范

1. `Read` 精确行号范围
2. old_string 覆盖足够上下文，确保唯一匹配
3. 删完读文件验证
4. Found 2+ matches → 停手，加大上下文重新匹配，不得 replace_all

## 六、定稿后九步对齐清单（每步必执行）

| 步骤 | 操作              | 文件位置                      |
| --- | --------------- | --------------------------- |
| 1   | 章节进度总表 → ✅定稿   | 设定库 `chapter-progress`      |
| 2   | 章节统计总览 → 已定稿数+1 | 设定库 `stats`                |
| 3   | 最后更新日期         | 设定库 `meta`                 |
| 4   | 前情摘要合并版        | `前情摘要合并版.md`              |
| 5   | 伏笔档案 → 已回收/已埋设  | 设定库 `foreshadowing`         |
| 6   | 人物设定库 → 新人物已添加  | 设定库 `personas`             |
| 7   | 物资消耗记录 → 已更新（如涉及） | 设定库 `supplies`            |
| 8   | 章节标题 ≥2 汉字      | 正文标题行                     |
| 9   | RAG 索引更新         | `.mcp/rag.py` + rag_index.json |

**汇报规则**：定稿后主动在同一条消息里逐项汇报完成状态。

---

## 七、去AI味禁用词

**程度副词**：非常、显著、大幅、极大地、深刻地、全面地、前所未有、根本性、彻底、高度、充分、深入、尤其、特别、相当、极其、格外、切实

**连接词**：此外、另外、同时、值得注意的是、需要指出的是、不仅如此、更重要的是、与此同时、除此之外、总的来说、综上所述、由此可见、不难看出、显而易见

**格式限制**：感叹号 ≤ 2 个、破折号 ≤ 1 次/千字、单句 ≤ 40 字

---

## 八、坑点记录

1. **禁止自己写正文** — 必须调用 MCP 工具
2. **禁止自己审稿** — 必须调用 MCP 工具
3. **文件存在 ≠ 流程已走完** — 根据用户指令判断
4. **patch 破折号导致句子断裂** — 替换后立即读文件验证
5. **未检查上章结尾就按大纲写** — 写新章前先读上章结尾 2-3 段，确认承接
6. **顾问建议含不存在的人名/地名** — grep 验证后再采纳
